Technical Requirement Document â€” Minimal "Hello World" Extension for Microsoft Dynamics 365 Business Central Document Purpose: Define requirements for a minimal AL extension for Business Central Version: 1.0 Date: 2026-05-02 Intended Audience: AL Developer, Tester, Business Reviewer 2. Brief Explanation / Overview This requirement defines a minimal per-tenant AL extension that extends the standard Customer table, adds one custom field, shows it on the Customer Card, and includes a small validation rule to demonstrate core Business Central extensibility. The solution must be delivered only through extension objects and must not directly modify Microsoft base objects. The design follows Microsoft's extension model utilizing tableextension and pageextension objects. 3. Objective To provide a developer-ready specification for a simple, non-intrusive extension that safely modifies the Dynamics 365 Business Central user interface and data schema while adhering to modern AL development best practices. 4. Scope The scope encompasses the creation of an AL extension containing a new custom field on the Customer table, exposure of this field on the Customer Card page, and implementation of basic validation logic ensuring minimum character length. 5. Functional Scope | Item | Requirement | |----------------------------|----------------------------------------------------------| | Standard entity to extend | Customer | | New field | PTE Hello World Note | | Field type | Text[100] | | UI exposure | Add field to Customer Card via page extension | | Business logic | Validate non-blank value must be at least 3 characters | | Constraint | No direct modification of base objects | 6. Developer Notes For a minimal, realistic per-tenant sample, this document utilizes a "PTE" prefix and object IDs within the 50000â€“99999 range. This range is structurally suitable for per-tenant customizations and isolated scenarios as per Microsoft's upgrade-safe extension model. Note: If this extension is intended for Microsoft AppSource in the future, the PTE prefix must be replaced with a registered 3-character affix, and the object ID range must match the assigned app object range. 7. Assumptions and Dependencies - The target environment is Dynamics 365 Business Central (cloud or on-premises running a recent runtime). - The developer has access to an AL development environment (Visual Studio Code with the AL Language extension). - The tester has access to a Sandbox/UAT environment for installation and validation of the extension. 8. AL Code Design TableExtension Design The solution uses a tableextension to add the new field to the Customer table and to host the validation trigger. tableextension 50100 "PTE Customer Ext" extends Customer { fields { field(50100; "PTE Hello World Note"; Text[100]) { Caption = 'Hello World Note'; DataClassification = CustomerContent; trigger OnValidate() begin if "PTE Hello World Note" = '' then exit; if StrLen("PTE Hello World Note") < 3 then Error('Hello World Note must contain at least 3 characters'); end; } } } PageExtension Design The solution requires a pageextension to display the field cleanly at the end of the General FastTab on the Customer Card. pageextension 50110 "PTE Customer Card Ext" extends "Customer Card" { layout { addlast(General) { field("PTEHelloWorldNote"; Rec."PTE Hello World Note") { ApplicationArea = All; Caption = 'Hello World Note'; ToolTip = 'Specifies a short demo note used to demonstrate Business Central extensibility.'; } } } } Optional Business Logic Variant If testing interaction is desired, the trigger can be extended with a notification message: trigger OnValidate() begin if Rec."PTE Hello World Note" = '' then exit; if StrLen(Rec."PTE Hello World Note") < 3 then Error('Hello World Note must contain at least 3 characters'); Message('Hello %1. The note has been updated.', Rec.Name); end; Recommendation: The Message popup is optional demo behavior only. For production-oriented implementations, keep only the Error(...) validation to prevent intrusive popups during high-volume data entry. 9. Requirement Mapping The tableextension object satisfies the requirement to extend the standard Customer table, adds exactly one custom field, and places the business validation rule in the table layer. This guarantees validation applies consistently regardless of the entry point (UI, API, or codeflow). The pageextension exposes the custom field on the Customer Card within the existing General fast tab, satisfying UI exposure without altering the base page directly. 10. Naming Conventions and Object ID Strategy All customized standard objects and newly created fields must adhere to the configured prefix (PTE). Identifiers must be kept within the 50000â€“99999 tenant customization range to ensure isolation and upgrade-safe alignment with AL extension best practices. 11. Deployment Notes The extension must be packaged as an .app file. Initial deployment will target a Sandbox environment to conduct User Acceptance Testing (UAT). Following UAT sign-off, the compiled extension can be uploaded directly to the Production tenant via the Extension Management page. 12. Build and Validation Expectations The AL codebase must compile cleanly without warnings or runtime errors. Base objects must not show any direct modifications. Validation logic must fire accurately for UI-driven inserts/updates. 13. Acceptance Criteria - A new AL extension is created without changing any base Microsoft object directly. - The Customer table is extended with one custom field named PTE Hello World Note. - The field is visible on the Customer Card page. - Entering a non-blank value shorter than 3 characters raises an error. - The code compiles cleanly and uses consistent naming and object IDs. 14.
