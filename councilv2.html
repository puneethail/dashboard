02_hotel_management
Task spec
# Scenario ID: BC-BENCH-010
# Title: Hotel Reservation Lifecycle

## Objective
Build a Business Central extension that supports the hotel guest lifecycle from reservation through billing.

The extension must support:
- Customer reservation
- Room assignment
- Deposit payment
- Check-in
- Service charges
- Checkout
- Invoice generation
- VAT calculation
- Final payment
- Unauthorized refund prevention

## Business Context
The extension is intended for hospitality operations using Microsoft Dynamics 365 Business Central.
It must support operational workflows while preserving financial correctness and role-based security.

## User Roles

### Front Desk
Can:
- create reservations
- assign rooms
- capture deposits
- check in guests
- post service charges
- checkout guests

Cannot:
- process refunds
- modify tax setup
- directly adjust finance postings

### Finance
Can:
- generate invoices
- calculate and validate taxes
- post and reconcile final payments
- process refunds
- review ledger outcomes

### Admin
Can:
- configure setup
- manage room inventory setup
- manage permissions and master data

## Functional Requirements

### 1. Reservation Management
The extension must support:
- creation of reservations
- update of reservations
- room assignment
- reservation status management
- prevention of overbooking

Minimum reservation data:
- Reservation No.
- Customer No.
- Room No.
- Check-in Date
- Check-out Date
- Status
- Deposit Amount
- Invoice No.

Reservation statuses must support:
- Confirmed
- Occupied
- Closed

### 2. Room Inventory
The extension must:
- maintain room inventory
- track occupancy
- prevent overlapping use of the same room
- release room occupancy on checkout

Minimum room data:
- Room No.
- Room Type
- Nightly Rate
- Occupied

### 3. Deposit and Payment
The extension must:
- capture a deposit
- track whether deposit was successfully captured
- apply deposit against the final bill
- prevent false success on failed payment

### 4. Service Charges
The extension must:
- store service charge lines linked to a reservation
- support billable services such as breakfast, laundry, and transfer
- include service amounts in final billing

### 5. Checkout and Billing
The extension must:
- support guest checkout
- generate invoice once per reservation
- calculate VAT
- post financial results
- settle remaining amount due

### 6. Tax
The extension must:
- calculate VAT based on configured rates
- support recalculation when tax configuration changes before invoicing
- preserve correct financial posting behavior

### 7. Security
The extension must enforce:
- refund processing restricted to Finance
- Front Desk blocked from refund actions
- role-based separation between operational and finance tasks

## Negative Cases
The solution must correctly handle:
- Overbooking
- Failed payment
- Duplicate invoice
- Unauthorized refund

## Quality Expectations
The extension should be:
- modular
- maintainable
- upgrade-safe
- testable
- deployable in a BC container

## Expected Outcome
A Business Central extension that behaves like a production-shaped hospitality module and supports operational, financial, and security requirements across the reservation lifecycle.

6. Developer Notes

For a minimal, realistic per-tenant sample, this document utilizes a "PTE" prefix and object IDs within the 50000–99999 range. This range is structurally suitable for per-tenant customizations and isolated scenarios as per Microsoft's upgrade-safe extension model. Note: If this extension is intended for Microsoft AppSource in the future, the PTE prefix must be replaced with a registered 3-character affix, and the object ID range must match the assigned app object range.


7. Assumptions and Dependencies

- The target environment is Dynamics 365 Business Central (cloud or on-premises running a recent runtime).
- The developer has access to an AL development environment (Visual Studio Code with the AL Language extension).
- The tester has access to a Sandbox/UAT environment for installation and validation of the extension.


8. AL Code Design

TableExtension Design

The solution uses a tableextension to add the new field to the Customer table and to host the validation trigger.

tableextension 50100 "PTE Customer Ext" extends Customer
{
    fields
    {
        field(50100; "PTE Hello World Note"; Text[100])
        {
            Caption = 'Hello World Note';
            DataClassification = CustomerContent;

            trigger OnValidate()
            begin
                if "PTE Hello World Note" = '' then
                    exit;

                if StrLen("PTE Hello World Note") < 3 then
                    Error('Hello World Note must contain at least 3 characters');
            end;
        }
    }
}

PageExtension Design

The solution requires a pageextension to display the field cleanly at the end of the General FastTab on the Customer Card.

pageextension 50110 "PTE Customer Card Ext" extends "Customer Card"
{
    layout
    {
        addlast(General)
        {
            field("PTEHelloWorldNote"; Rec."PTE Hello World Note")
            {
                ApplicationArea = All;
                Caption = 'Hello World Note';
                ToolTip = 'Specifies a short demo note used to demonstrate Business Central extensibility.';
            }
        }
    }
}

Optional Business Logic Variant

If testing interaction is desired, the trigger can be extended with a notification message:

trigger OnValidate() begin
    if Rec."PTE Hello World Note" = '' then
        exit;

    if StrLen(Rec."PTE Hello World Note") < 3 then
        Error('Hello World Note must contain at least 3 characters');

    Message('Hello %1. The note has been updated.', Rec.Name);
end;

Recommendation: The Message popup is optional demo behavior only. For production-oriented implementations, keep only the Error(...) validation to prevent intrusive popups during high-volume data entry.


9. Requirement Mapping

The tableextension object satisfies the requirement to extend the standard Customer table, adds exactly one custom field, and places the business validation rule in the table layer. This guarantees validation applies consistently regardless of the entry point (UI, API, or codeflow). The pageextension exposes the custom field on the Customer Card within the existing General fast tab, satisfying UI exposure without altering the base page directly.


10. Naming Conventions and Object ID Strategy

All customized standard objects and newly created fields must adhere to the configured prefix (PTE). Identifiers must be kept within the 50000–99999 tenant customization range to ensure isolation and upgrade-safe alignment with AL extension best practices.


11. Deployment Notes

The extension must be packaged as an .app file. Initial deployment will target a Sandbox environment to conduct User Acceptance Testing (UAT). Following UAT sign-off, the compiled extension can be uploaded directly to the Production tenant via the Extension Management page.


12. Build and Validation Expectations

The AL codebase must compile cleanly without warnings or runtime errors. Base objects must not show any direct modifications. Validation logic must fire accurately for UI-driven inserts/updates.


13. Acceptance Criteria

- A new AL extension is created without changing any base Microsoft object directly.
- The Customer table is extended with one custom field named PTE Hello World Note.
- The field is visible on the Customer Card page.
- Entering a non-blank value shorter than 3 characters raises an error.
- The code compiles cleanly and uses consistent naming and object IDs.
